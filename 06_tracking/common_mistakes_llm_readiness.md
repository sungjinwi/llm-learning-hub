# Common Mistakes For LLM Readiness

작성일: 2026-05-20

이 문서는 LLM 활용 과정 전에 특히 조심해야 할 흔한 실수와 그 이유를 정리한다.

## 1. Accuracy가 높으면 좋은 모델이라고 생각함

왜 실수인가:

데이터가 불균형하면 accuracy는 쉽게 착시를 만든다. 예를 들어 전체 데이터의 95%가 정상이고 5%만 이상 거래라면, 모든 데이터를 정상이라고 예측해도 accuracy는 95%가 된다. 하지만 이 모델은 정작 찾아야 하는 이상 거래를 하나도 찾지 못한다.

대신 확인할 것:

- precision
- recall
- F1
- confusion matrix
- 실제 서비스에서 더 치명적인 오류가 무엇인지

LLM/RAG에서도 비슷하다. 답변이 그럴듯해 보이는 비율보다, 근거 문서를 제대로 찾았는지와 중요한 질문에서 틀리지 않는지가 더 중요하다.

## 2. 코드를 실행만 하고 출력 예측을 하지 않음

왜 실수인가:

실행 결과만 보면 맞은 것처럼 느껴지지만, 실제로는 왜 그렇게 나왔는지 모를 수 있다. 출력 예측을 먼저 해야 변수, 조건문, 반복문, 함수 호출 흐름을 머릿속으로 추적하는 능력이 생긴다.

대신 확인할 것:

- 실행 전에 출력 예상하기
- 예상과 실제 출력이 다르면 어느 줄에서 생각이 틀렸는지 찾기
- 작은 입력으로 다시 테스트하기

## 3. 에러 메시지를 읽지 않고 바로 AI에게 붙여넣음

왜 실수인가:

에러 메시지에는 파일명, 줄 번호, 에러 종류가 들어 있다. 이 정보를 읽지 않으면 같은 문제를 반복해서 AI에게 의존하게 된다.

대신 확인할 것:

- 에러 종류: `NameError`, `TypeError`, `KeyError`, `IndexError`
- 에러가 난 줄
- 그 줄에서 사용한 변수와 데이터 타입
- 최소 재현 코드

## 4. list, dict, DataFrame, tensor를 전부 비슷한 표로 생각함

왜 실수인가:

각 자료구조는 접근 방식과 연산 방식이 다르다. LLM 서비스에서는 문서 목록은 `list`, 문서 metadata는 `dict`, 표 데이터는 `DataFrame`, 모델 입력은 tensor처럼 역할이 나뉜다. 이를 구분하지 못하면 LangChain이나 Pandas 코드에서 데이터 흐름을 잃기 쉽다.

대신 확인할 것:

- list: 순서와 인덱스
- dict: key-value
- DataFrame: 행과 열
- tensor: shape과 dtype

## 5. API 응답을 문자열처럼만 다룸

왜 실수인가:

LLM API 응답은 보통 중첩된 JSON 구조다. 필요한 값이 어디에 들어 있는지 모르면 응답 파싱, 로그 저장, 오류 처리가 어렵다.

대신 확인할 것:

- 응답 전체 구조 출력하기
- 필요한 필드 경로 찾기
- 없는 key에 접근할 때의 예외 처리
- 요청 JSON과 응답 JSON을 따로 저장하기

## 6. API key를 코드에 직접 적음

왜 실수인가:

API key가 GitHub, 노트북, 스크린샷, 로그에 노출될 수 있다. 노출되면 비용이 발생하거나 계정이 제한될 수 있다.

대신 확인할 것:

- `.env` 사용
- `.gitignore`에 `.env` 추가
- 로그에 key가 찍히지 않게 하기
- key를 공유 문서에 붙여넣지 않기

## 7. Prompt를 감으로만 고침

왜 실수인가:

prompt를 계속 바꾸면서 결과만 보면 무엇이 좋아졌는지 알 수 없다. 같은 질문 세트로 비교하지 않으면 개선이 아니라 우연일 수 있다.

대신 확인할 것:

- prompt 버전 기록
- 고정된 테스트 질문 5개 이상
- 기대 답변 기준
- 실패한 답변을 별도 기록

## 8. RAG에서 문서를 많이 넣으면 무조건 좋아진다고 생각함

왜 실수인가:

문서가 많아지면 검색 대상이 넓어져 관련 없는 chunk가 섞일 가능성도 커진다. chunk 크기, overlap, metadata, 검색 쿼리, top-k 설정이 맞지 않으면 답변 품질이 오히려 나빠진다.

대신 확인할 것:

- 질문에 대해 실제 검색된 chunk 확인
- 검색 결과가 답변에 충분한지 확인
- chunk 크기와 overlap 바꿔 비교
- top-k를 바꿔 비교

## 9. RAG 실패를 LLM 문제로만 봄

왜 실수인가:

RAG는 검색과 생성이 합쳐진 시스템이다. 답변이 틀렸을 때 원인이 검색 실패인지, prompt 실패인지, 모델 생성 실패인지 분리하지 않으면 고칠 수 없다.

대신 확인할 것:

- retrieval 결과가 맞는가?
- 검색된 문서 안에 정답 근거가 있는가?
- prompt가 근거만 사용하라고 지시하는가?
- 모델이 근거 없는 내용을 추가했는가?

## 10. LangChain 코드를 블랙박스로 사용함

왜 실수인가:

LangChain은 편하지만 추상화가 많다. 입력과 출력 타입을 모르면 에러가 났을 때 어느 단계가 문제인지 찾기 어렵다.

대신 확인할 것:

- chain의 입력 타입
- chain의 출력 타입
- retriever가 반환하는 문서 객체 구조
- prompt에 실제로 들어간 문자열
- 각 단계 사이에 `print`나 로그 추가

## 11. 대화 이력 memory를 무조건 많이 넣음

왜 실수인가:

대화 이력이 길어지면 비용과 latency가 증가하고, 오래된 정보가 현재 질문을 방해할 수 있다. context window를 넘으면 일부 내용이 잘리거나 중요한 지시가 희석된다.

대신 확인할 것:

- 최근 대화만 넣기
- 요약 memory 사용
- 중요한 사용자 정보와 일시적 대화를 분리
- 토큰 사용량 확인

## 12. UI를 먼저 만들고 핵심 로직 검증을 나중에 함

왜 실수인가:

Gradio 같은 UI는 빠르게 만들 수 있지만, 검색과 답변 로직이 검증되지 않은 상태에서 UI를 붙이면 문제 원인을 찾기 어려워진다.

대신 확인할 것:

- 먼저 Python 함수로 입력과 출력 검증
- 그 다음 API 또는 RAG 연결
- 마지막에 UI 연결
- UI에는 최소한의 상태만 넣기

## 13. AI가 준 코드를 이해하지 않고 누적함

왜 실수인가:

처음에는 빨라 보이지만, 코드가 길어질수록 수정이 불가능해진다. 특히 LLM 프로젝트는 API, prompt, retriever, chain, UI가 연결되므로 한 부분을 이해하지 못하면 전체가 막힌다.

대신 확인할 것:

- AI가 준 코드의 입력, 처리, 출력을 직접 설명하기
- 모르는 줄에 주석 달기
- 한 번에 한 기능만 추가하기
- 작동하는 버전을 따로 저장하기

## 14. 비용과 속도를 나중 문제로 미룸

왜 실수인가:

LLM 서비스는 호출할 때마다 비용과 시간이 발생한다. 개발 중에는 작아 보여도 반복 테스트, 긴 context, 큰 top-k, 여러 모델 호출이 누적되면 비용과 latency가 커진다.

대신 확인할 것:

- 호출 횟수 기록
- 입력과 출력 토큰 대략 확인
- 캐싱 가능 여부 확인
- 모델 크기와 품질의 trade-off 비교

## 15. 결과가 그럴듯하면 근거 확인을 생략함

왜 실수인가:

LLM은 틀린 답도 자연스럽게 말할 수 있다. 특히 법률, 금융, 기업 분석처럼 과정에서 다루는 주제는 근거 확인이 중요하다.

대신 확인할 것:

- 답변에 사용된 문서 chunk 표시
- 근거가 없는 답변은 모른다고 처리
- 출처 문장과 답변 문장을 비교
- 중요한 답변은 사람이 검토하는 단계 추가
